SCENARIO 01
New Employee Onboarding — Create User Account
Beginner
🎫 TICKET #1001 — HR REQUEST
New starter joining Monday — need AD account created
"New employee Sarah Johnson starting Monday. Department: Finance. Job title: Accounts Payable. Needs access to Finance shared drive. Please create account before 9am Monday." — HR Team
New-ADUser Add-ADGroupMember ADUC PowerShell
1

Created Finance OU in ADUC

Right clicked lab.local → New → Organizational Unit → Finance

2

Created user account via PowerShell

New-ADUser -Name "Sarah Johnson" -SamAccountName "sarah.johnson" `
  -Department "Finance" -Title "Accounts Payable" `
  -Path "OU=Finance,OU=_COMPANY,DC=lab,DC=local" `
  -AccountPassword (ConvertTo-SecureString "Welcome@2025!" -AsPlainText -Force) `
  -ChangePasswordAtLogon $true -Enabled $true
3

Added to Finance security group

Add-ADGroupMember -Identity "GRP-Finance" -Members "sarah.johnson"
4

Verified login on CLIENT-01

Logged into CLIENT-01 as LAB\sarah.johnson — confirmed login and password change prompt.

💼 Real World: Most companies have HR systems that auto-trigger IT tickets for new starters. Standard onboarding checklist — AD account, email, software access, VPN setup.

📸 Portfolio Evidence — Screenshots Taken

  • PowerShell command running successfully — user created
  • User visible in ADUC under Finance OU
  • Group membership showing GRP-Finance
  • Successful login on CLIENT-01 as sarah.johnson

2

SCENARIO 07
Bulk User Creation — New Department Onboarding
Intermediate
🎫 TICKET #1007
10 new call centre staff starting Monday
"We have 10 new call centre agents all starting Monday. HR sent a spreadsheet. Need all accounts created by Friday." — Operations Manager
Import-CSVForEach-ObjectPowerShell Automation
1

Created CSV file from HR spreadsheet

FirstName,LastName,Department,Title
James,Wilson,CallCentre,Agent
Emma,Brown,CallCentre,Agent
Liam,Taylor,CallCentre,Agent
2

Ran bulk creation script

Import-Csv "C:\users.csv" | ForEach-Object {
  $username = ($_.FirstName[0] + $_.LastName).ToLower()
  New-ADUser -Name "$($_.FirstName) $($_.LastName)" -SamAccountName $username `
    -Enabled $true -ChangePasswordAtLogon $true
  Write-Host "Created: $username"
}
3

Verified all users created

Get-ADUser -Filter * -SearchBase "OU=CallCentre,OU=_COMPANY,DC=lab,DC=local" | Select-Object Name
💼 Real World: PowerShell automation is a highly valued skill. Creating 10 accounts manually takes 30+ minutes. This script does it in seconds.

SCENARIO 10
Security Audit — Find and Disable Inactive Accounts
Advanced
🎫 TICKET #1010 — SECURITY AUDIT
Annual audit — find and disable stale accounts
"Annual security audit requires identifying and disabling all AD accounts unused for 90+ days. Compliance requirement." — Security Manager
Get-ADUserExport-CsvDisable-ADAccountAD Audit
1

Found inactive accounts (never logged in)

Get-ADUser -Filter * -Properties LastLogonDate, Department |
  Where-Object {$_.LastLogonDate -eq $null -and $_.Enabled -eq $true} |
  Select-Object Name, SamAccountName, LastLogonDate, Department
2

Exported audit report to CSV

Export-Csv "C:\InactiveUsers_$(Get-Date -f 'yyyy-MM-dd').csv" -NoTypeInformation
3

Disabled all inactive accounts

Disable-ADAccount -Identity $_
Set-ADUser -Identity $_ -Description "Disabled — Inactive 90 days — $(Get-Date -f 'yyyy-MM-dd')"
💼 Real World: Ghost accounts (ex-staff still active) are a major security risk. Companies with compliance requirements run these audits quarterly.

📸 Portfolio Evidence — Screenshots Taken

  • Inactive accounts found in PowerShell results
  • CSV report generated and opened in Excel
  • Accounts disabled with description audit trail
SCENARIO 11
Fine-Grained Password Policy for IT Admins
Advanced
🎫 TICKET #1011 — SECURITY
IT Admin accounts need stricter password policy
"Following penetration test, IT Admin accounts need stricter password policy — min 16 chars, change every 30 days, lock after 3 attempts." — CTO
Fine-Grained PolicyPSOADAC
1

Opened ADAC and navigated to Password Settings Container

ADAC → lab (local) → System → Password Settings Container

2

Created PSO-ITAdmins policy

New-ADFineGrainedPasswordPolicy -Name "PSO-ITAdmins" -Precedence 10 `
  -MinPasswordLength 16 -MaxPasswordAge "30.00:00:00" `
  -LockoutThreshold 3 -LockoutDuration "00:30:00" `
  -PasswordHistoryCount 10 -ComplexityEnabled $true
3

Applied to GRP-IT-Admins

Add-ADFineGrainedPasswordPolicySubject -Identity "PSO-ITAdmins" -Subjects "GRP-IT"
4

Verified policy applying to IT admin

Get-ADUserResultantPasswordPolicy -Identity "bob.it"
💼 Real World: Privileged account management — admin accounts are high value targets and need stronger protection than regular user accounts.

📸 Portfolio Evidence — Screenshots Taken

  • PSO created in Password Settings Container (ADAC)
  • Policy applied to GRP-IT-Admins
  • Resultant policy showing on bob.it account
SCENARIO 12
Troubleshoot — Access Denied After Internal Transfer
Advanced
🎫 TICKET #1012
Can log in but getting Access Denied on Finance drive
"I transferred from HR to Finance last week. Can log in fine but Finance drive shows Access Denied. Still see HR drive which I shouldn't have anymore." — Alice
TroubleshootingGroup MembershipMove-ADObject
1

Diagnosed — checked group memberships

Get-ADPrincipalGroupMembership -Identity "alice.hr" | Select-Object Name
# Found: still in GRP-HR, NOT in GRP-Finance — root cause identified
2

Removed from old HR group

Remove-ADGroupMember -Identity "GRP-HR" -Members "alice.hr" -Confirm:$false
3

Added to Finance group

Add-ADGroupMember -Identity "GRP-Finance" -Members "alice.hr"
4

Moved user to Finance OU

Move-ADObject -Identity "CN=Alice HR,OU=HR,OU=_COMPANY,DC=lab,DC=local" `
  -TargetPath "OU=Finance,OU=_COMPANY,DC=lab,DC=local"
5

Verified with whoami after re-login

whoami /groups # confirmed GRP-Finance present, GRP-HR removed
💼 Real World: Internal transfers are a common source of permissions issues. Check group memberships first when someone gets Access Denied — 80% of the time that's the cause.
